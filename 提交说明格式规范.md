# Git 提交说明格式规范

## 基本原则

- 提交说明应清晰、简洁地描述本次提交的内容
- 使用中文编写
- 遵循一致的格式和结构

## 格式要求

### 标签格式

提交信息以标签开头，使用方括号 `[]` 包裹，常用标签包括：

| 标签 | 说明 | 使用场景 |
|------|------|----------|
| `[INIT]` | 项目初始化 | 首次提交或项目初始化 |
| `[FEAT]` | 新功能 | 添加新特性或功能 |
| `[FIX]` | 修复问题 | 修复 bug 或错误 |
| `[REFACTOR]` | 重构 | 代码重构，不改变功能 |
| `[DOCS]` | 文档 | 文档更新或新增 |
| `[STYLE]` | 样式 | 代码格式调整（不影响功能） |
| `[TEST]` | 测试 | 添加或修改测试 |
| `[CHORE]` | 杂项 | 构建过程、工具链等 |
| `[PERF]` | 性能优化 | 性能提升相关 |
| `[REMOVE]` | 删除 | 删除功能或文件 |
| `[UPDATE]` | 更新 | 更新依赖或配置 |

#### 复合标签

当一次提交涉及多个类型的更改时，可以使用复合标签。多个标签用逗号分隔，放在同一个方括号内：

**格式：** `[标签1,标签2,标签3] 简短描述`

**常用复合标签组合：**

| 组合 | 说明 | 使用场景 |
|------|------|----------|
| `[FEAT,DOCS]` | 新功能 + 文档 | 添加新功能同时更新文档 |
| `[FIX,TEST]` | 修复 + 测试 | 修复问题并添加测试用例 |
| `[REFACTOR,DOCS]` | 重构 + 文档 | 重构代码并更新相关文档 |
| `[FEAT,STYLE]` | 新功能 + 样式 | 添加新功能同时调整代码风格 |
| `[UPDATE,CHORE]` | 更新 + 杂项 | 更新依赖和构建配置 |
| `[FIX,PERF]` | 修复 + 性能优化 | 修复问题同时提升性能 |
| `[FEAT,DOCS,TEST]` | 新功能 + 文档 + 测试 | 完整的功能开发 |

### 提交格式

**单一标签格式：**

```
[标签] 简短描述

详细描述（可选）
```

**复合标签格式：**

```
[标签1,标签2] 简短描述

详细描述（可选）
```

#### 示例

**单一标签示例：**

```
[FEAT] 添加 Kotlin 原生支持

- 集成 Kotlin 编译器
- 添加 Kotlin Native 支持
- 更新构建脚本
```

```
[FIX] 修复内存泄漏问题

修复了在 Android 10+ 设备上出现的内存泄漏问题，
原因是 Activity 未正确释放资源。
```

```
[REFACTOR] 优化代码结构

将工具类从 MainActivity.kt 中分离到独立的 Utils.kt 文件，
提高代码可维护性。
```

**复合标签示例：**

```
[FEAT,DOCS] 添加完整的工作区管理功能

- 添加工作区管理界面
- 添加文件树视图和编辑器界面
- 实现文件管理功能：新建、重命名、删除
- 支持文件夹展开/折叠和子文件夹操作
- 添加行号显示功能
- 更新 README 文档
```

```
[FIX,TEST] 修复登录验证问题并添加测试用例

修复了用户登录时验证码校验失败的问题。
同时添加了完整的单元测试覆盖边界情况。

- 修复：验证码大小写敏感问题
- 测试：添加 15 个测试用例
- 文档：更新 API 说明
```

```
[REFACTOR,DOCS,PERF] 优化数据库查询性能并更新文档

- 重构：使用索引优化查询性能
- 性能：查询速度提升 40%
- 文档：更新数据库设计文档和 API 说明
```

## 注意事项

1. **简短描述**：使用祈使句，首字母大写，不超过 50 个字符
2. **详细描述**：如需详细说明，使用多行，每行不超过 72 个字符
3. **空行**：简短描述和详细描述之间空一行
4. **单一职责**：每个提交只做一件事
5. **避免合并提交**：除非必要，避免使用 merge 提交

## 分支命名规范

- `main` - 主分支，稳定版本
- `dev` - 开发分支
- `feature/功能名称` - 功能开发分支
- `fix/问题描述` - 问题修复分支
- `refactor/重构内容` - 重构分支

## 工作流程

1. 从主分支创建功能分支：`git checkout -b feature/xxx`
2. 在功能分支上开发并提交
3. 推送到远程：`git push origin feature/xxx`
4. 创建 Pull Request 或合并到主分支
